# viMarkdown エディタ・プレビューカーソル位置同期自動テストを実装してみた
## アウトライン
### プレビューでの編集機能がついつい欲しくなった
- viMarkdown: ライブプレビューでマークダウンが正しいかどうかが瞬時にわかるので、マークダウン初心者でも安心
- マークダウンソースより、プレビューの方が文章を読みやすい
- ついついプレビューで文字挿入・削除してしまい、何も変化が無いので、あれってなることが多々あった
### 我慢できずにプレビューでの編集機能をついつい実装開始してしまった
- 典型的な状況（正常系）では、正しく編集が短時間で可能にった
- が、想定していない状況（エッジケース）が多々あり、
- すべての状況に対応するのに、膨大なQAコストを費やした
  - 工数が膨大なことはわかっていたのだが、
  - プレビュー側で文字挿入・削除された場合は、プレビューカーソル位置に対応するエディタカーソル位置に同じ文字を挿入・削除すればよい
  - なので、PtoE カーソル位置同期処理が必要
  - 膨大な数の問題が発生し、副作用も少なくなかった
### 膨大な数の問題が出るので同期処理自動テストをついつい実装した
   - 片方のカーソルを先頭から末尾まで順に移動させ、もう片方が正しい位置に移動したかをチェック
   - 正しい位置に移動したかをどうやってチェックする？
   - テストケースの各位置に対する正しい位置を手動で用意するのは現実的でない
   - 生成AIにテストケース＆正解位置を生成してもらうのも考えたが、いまいち信用できなかった
   - 異なるアルゴリズムにより対応位置を計算し比較
     - 本来の同期アルゴリズム：検索ベース
     - 自動テスト：（非表示の行・文字をスキップする）行番号・カラムベース
- テストに失敗した場合、本体の同期アルゴリズムの問題なのか、自動テストの方にバグがあるのか判定が必要
  - NASA宇宙船では複数のコンピュータが同じアルゴリズムで計算し、多数決で制御を行っていた（宇宙線による誤動作を防ぐため）
  - viMarkdown では人間（作者）が判定
### 結果
- 自動テストにより膨大な問題が発見された
- あとどの程度問題対処すればRC版に移行できるか予測可能
  - ゴールは見えなくても、ゴールまでの距離が推測可能になった
- 副作用発生がすぐに検知できる
